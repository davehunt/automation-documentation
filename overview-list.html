Below you will find a list of the various automated testing frameworks in use at Mozilla, along with a brief description and a link to read more. Generally, you should pick the lowest-level framework that you can. If you are testing JavaScript but don't need a window, use XPCShell. If you're testing page layout, try to use reftest. The advantage is that you don't drag in a lot of other components that might have their own problems, so you can hone in quickly on any bugs in what you are specifically testing.

If you have questions, join #automation on irc.mozilla.org or post to the public mailing list.

AutoPhone

AutoPhone is a platform for running automated tests on physical mobile devices--phones and tablets.

Compiled-Code

Written in C++, compiled-code tests can test pretty much anything but are difficult to write properly.

Crashtest

If you've found a page that crashes Firefox, add a crashtest to make sure future versions don't experience this crash again.

Eideticker

Eideticker is an automated test harness that captures and analyzes browser output via HDMI.

Firebug

The Firebug Test Runner intends to automate the process of running the Firebug tests.

JMeter

Use JMeter for load testing functional behavior and measuring performance.

JS Reftest

JS Reftest is used to run the JavaScript tests in the browser using the reftest framework.

JS Shell

JS Shell tests are specifically for the JavaScript engine. They test every piece of the engine.

Marionette

Marionette is a test automation framework used to drive the UI and JS/XPCOM layer of remote or local instances.

Mochitest

A11y

A11y tests accessibility interfaces.

Browser Chrome

Browser Chrome runs in the scope of the browser window, this is a rough UI automation tool testing how the browser interacts with itself and with content.

Chrome

Chrome tests run with high privileges that can test a lot of the browser's functionality. Tests that verify JavaScript interactions with chrome-level objects should go here.

IPC

IPC tests plugin APIs, particularly out-of-process plugins.

Plain

Plain should be used to test DOM APIs and other pieces of functionality exposed to web content (i.e. requiring no special permissions).

Robocop

Robocop tests are designed for testing the native UI of Android devices by sending events to the front end.

Mozmill

Mozmill is an extensive framework for browser automation.

Peptest

Peptest measures responsiveness, how "snappy" Firefox/Thunderbird feels, by issuing alerts when the event loop is stuck for more than 50 ms.

Reftest

The reftest harness compares the display of two web pages.

Selenium

Selenium is a suite of tools for browser automation suitable for testing web content across multiple browsers.

SpeedTests

SpeedTests is a framework for executing arbitrary tests, which report results via JavaScript calls, in several browsers.

Talos

Talos is a framework for performance testing.

TPS

TPS is a multi-profile test automation framework for Firefox Sync.

XPCShell

XPCShell tests are console JavaScript tests. There is no chrome, no content, no window. XPCShell is useful for testing low-level things, such as XPCOM components.
